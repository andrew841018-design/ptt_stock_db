## Big picture

```
Before (ETL):
  Scrapers -> Pydantic validation + date parse + PII hash (Python) -> OLTP (articles)
  dbt staging = thin CAST -> marts

After (EL+T):
  Scrapers -> raw_articles (JSONB, append-only) -> dbt staging cleans -> intermediate -> marts
```

All transformation logic moves to SQL/Jinja. Lineage becomes visible via `dbt docs generate`.
Python keeps **extract** and **load-to-raw** only.

### Open decisions (resolved 2026-05-12 by Andrew + Claude)

1. **Raw layer**: ✅ **PG** confirmed (Andrew). Existing infra, no S3 cost.
2. **Surrogate `article_id`**: ⚠️ **SERIAL int** chosen by Andrew (override). Phase 1 keeps `sha256(url)` hex for verification (disposable). **Phase 2 must switch** to `MIN(raw_articles.id) OVER (PARTITION BY url ORDER BY ingested_at ASC)` per URL group → stable int per canonical row, FK-compatible with OLTP `articles.article_id` during cutover window. See Phase 2 step 5b below.
3. **`dbt_utils`**: ✅ keep (industry standard, mature, will use in Phase 2/3 for `equal_rowcount`, `mutually_exclusive_ranges`, etc.).
4. **PII hash 16-char**: ✅ keep (Claude). Storage savings 35万 × 48B ≈ 17 MB; collision risk negligible (10⁻¹¹ at unique author count ≈ 10K); upgrading to 64 would break cutover hash parity with OLTP.
5. **Date parse strategy**: ⚠️ Phase 1 ISO-only stays. **Phase 2 must add `parse_date_multi_format(raw_date, source_name)` dbt macro** — scrapers MUST NOT normalize before writing raw (that violates EL+T). Macro dispatches per source:
   - PTT: Taiwan timezone `YYYY-MM-DD HH:MM:SS`
   - CNN / WSJ / MarketWatch RSS: RFC 2822 (`%a, %d %b %Y %H:%M:%S %Z`)
   - Reddit: Unix epoch seconds (cast to bigint, `to_timestamp()`)
   - yfinance / Wayback / cnyes: ISO 8601
   - Fallback: NULL + log to dbt singular test for surfacing parse failures

#### Phase 2b risk

MEDIUM. Dual-write doubles INSERT load — measured at ~100 articles/hr cron, so +100 small JSONB inserts/hr is negligible (~10ms total). The risk surface is:
- Partial failure (_save_to_raw OK but v1 OLTP fails): raw row exists, no v1 row. dbt staging still sees the article; v1 API still queries the old row from before. Acceptable for dual-write (raw will be canonical post-Phase-3).
- Partial failure (_save_to_raw fails, v1 OLTP not attempted): no rows anywhere. Same failure mode as pre-Phase-2b. The contract preserves correctness.
- Comments / large payloads (PTT articles with 30+ comments): JSONB serialization can produce 50-100 KB payloads per article. PG handles JSONB up to 1 GB per row, so no risk.

#### Open decisions / known limitations

1. **wayback_backfill streaming raw write commits per-batch (25 rows) along with v1 row.** If v1 INSERT fails mid-batch, the raw row in the same transaction is also rolled back. This is INCONSISTENT with the base_scraper._save_to_db design (where _save_to_raw uses its own connection). The trade-off:
   - Pro: avoids 25 extra get_pg() context manager creations per commit batch
   - Con: tightly couples raw + v1 fate in wayback path

   Phase 3 will collapse this by deleting v1 path; wayback streaming will write only to raw_articles in one cursor.
2. **`_save_sentiment_scores_to_raw` is a no-op stub** — bert_sentiment.py is out of scope per task constraint. Phase 3 will wire it up (or replace the BERT path entirely via dbt-side LLM inference).
3. **Comments persistence** — PTT comments are written to BOTH OLTP `comments` table (v1) AND embedded in raw_articles.raw_payload (v2). v2 path's stg_articles_v2 ignores comments; downstream comment analytics still need v1 or a dedicated stg_comments_v2 model (deferred).
4. **Row-count parity** between fact_sentiment v1 and fact_sentiment_v2 will not match until raw_articles is backfilled from OLTP articles (Phase 3 step 1: `INSERT INTO raw_articles SELECT ... FROM articles`). In the current state, v1 has 380K articles' worth, v2 has 1005 PTT seed + new ingests only.

   INSERT INTO raw_articles (source_name, raw_payload, ingested_at)
   SELECT s.source_name,

              'url',          a.url,
              'title',        a.title,
              'author',       a.author,

          a.scraped_at

       SELECT (raw_payload->>'oltp_article_id')::int

       WHERE raw_payload ? 'oltp_article_id'

### Phase 2b': Outbox Pattern reframing + Mongo failure tolerance (DONE 2026-05-15)

#### Why this reframe

Phase 2b shipped the mechanics of dual-write. Phase 2b' is **the same mechanics framed
explicitly as the Outbox Pattern** plus a small defense-in-depth fix on the Mongo write
path. No architectural change to the existing dbt models, scrapers, or feature flag;
this is documentation + one try/except.

#### The Outbox Pattern (canonical framing)

Old ETL had a classic **dual-write problem**:

    Scraper (old):
      1. fetch HTTP
      2. write Mongo raw_responses    <-- system A
      3. validate + write PG articles <-- system B
         ^ steps 2+3 are NOT atomic — a crash between them desynchronizes A and B

Phase 2b' is shaped exactly like the Outbox Pattern:

    Scraper (Phase 2b'):
      1. fetch HTTP
      2. (best-effort) write Mongo raw_responses  <-- ARCHIVAL, may fail, no raise
      3. BEGIN TX:
           INSERT INTO raw_articles (source_name, raw_payload, ingested_at)  <-- THE OUTBOX
         COMMIT                                                              <-- atomic, durable
      4. (legacy) write OLTP articles / comments  <-- dual-write for cutover window

    PG raw_articles (outbox) = SOURCE OF TRUTH
      |
      v  (dbt acts as the "outbox poller / publisher")
    stg_articles_v2 -> int_articles_with_sentiment -> fact_sentiment_v2 / mart_daily_summary_v2

**Why this satisfies the Outbox Pattern**:

1. Scraper has ONE durable write that must succeed atomically: `_save_to_raw(articles)` does INSERT INTO raw_articles in a single transaction.
2. Mongo write is **best-effort archival**, NOT primary. `_store_raw()` now wraps `save_raw_response()` in try/except — Mongo downtime swallowed.
3. OLTP `articles` write is **legacy dual-write**, kept ONLY for cutover window. Phase 3 removes it.
4. dbt is the outbox consumer: stg_articles_v2 reads raw_articles, dedups via row_number() OVER (PARTITION BY url ORDER BY ingested_at DESC).

#### What Phase 2b' actually changed

1. `dependent_code/scrapers/base_scraper.py` — `_store_raw()` wrapped in try/except. Defense-in-depth against non-PyMongoError exceptions from MongoClient stack.
2. `MIGRATION_PLAN.md` — this section.

#### Layered contract

| Layer | System | Role | Failure tolerance |
|-------|--------|------|-------------------|
| Outbox (primary) | PG `raw_articles` | Source of truth for dbt EL+T | MUST succeed |
| Archival | Mongo `raw_responses` | HTTP body for repair/reparse | Best-effort; swallowed |

## Phase 3: Delete legacy ETL, rename _v2, drop old tables (NOT done)

### Goal

Single source of truth. No 'old' path. `stg_*_v2.sql` becomes `stg_*.sql`. Old OLTP gone.

### Concrete steps

2. Cut marts to read from intermediate (which reads from `stg_*_v2`).
3. Wait one cron cycle, verify mart row counts stable.
4. Remove `dependent_code/pii_masking.py` and `dependent_code/scraper_schemas.py` (the validation logic).
5. Remove `dbt/models/staging/stg_articles.sql stg_sentiment_scores.sql stg_sources.sql` and `_sources.yml`.
6. Rename `dbt/models/staging/stg_articles_v2.sql` to `stg_articles.sql` (plus update refs in fact_sentiment.sql).
7. DROP TABLE articles, sentiment_scores, sources CASCADE; (after verifying no app reads them. api.py, data_mart.py, dw_etl.py, QA.py, ge_validation.py, reparse.py, labeling_tool.py, llm_labeling.py, bert_sentiment.py, ai_model_prediction.py ALL touch these tables, full audit needed).
8. Update `_schema.yml` to drop `raw_ptt` prefix references, just `ptt`.
9. Delete `transform()`, `_step('pii')`, `_step('transform')` from pipeline.py.

### Phase 3 risk

HIGH. 10+ Python modules read articles/sentiment_scores/sources directly via SQL. Each must be migrated to query the new stg_articles/intermediate/mart views, OR be deleted (e.g., QA.py and ge_validation.py are arguably superseded by dbt tests).

### Phase 3 rollback

- Restore old tables from S3 backup (per backup.py).
- Revert the mart cutover.
- Re-deploy old scrapers from previous tag.

### Estimated effort

1-2 weeks. Bulk of the work is auditing every direct SQL reference to old OLTP tables and writing migration shims or refactoring callers.
